# WRITING.md

このファイルは、`200-Output`フォルダに格納されている記事・日記・メモの文体や構成の特徴を分析し、今後の執筆の指針とすることを目的としています。

---

## 全体共通ルール (Common Rules)

### Frontmatter

記事の先頭には必ずYAML形式のメタデータを付与する。

```yaml
---
tags: [日記]           # 配列形式。複数可: [技術, 日記]
aliases: []
created: 2026-01-23 20:00
updated: 2026-01-23 21:03
link: https://sizu.me/kmdkuk/posts/xxx  # 公開先URL（任意）
published: true        # 公開済みかどうか（任意）
---
```

### ファイル名

- 形式: `YYYYMMDDHHMM_タイトル（YYYY年MM月DD日）.md`
- 例: `202601232000_29歳になった。(2026年1月23日).md`
- タイトルにはキャッチーなフレーズを使うことが多い

### 署名

記事の最後には必ず日付と名前を入れる。

- 基本形式: `YYYY年MM月DD日 kouki`
- AIと協働した場合: `w/ChatGPT` や `w/Gemini` を付記
- 例: `2025年12月27日 kouki w/Gemini`

### 一人称

- 日記・エッセイ: 「自分」を基本とする
- 技術記事（Zenn等の外部公開）: 「筆者」「私」を使用

## カテゴリ別の書き方 (Category Styles)

### 1. 日記 (Diary)

**タグ**: `[日記]`

**文体**:

- 「だ・である」調
- カジュアルで口語的な表現も許容（「超うめぇ」「チマチマ」「ガッツリ」）
- 断定を避け、推測・保留の表現を多用

**構成**:

- 段落は短く、1〜2文で改行することが多い
- 余白を多めに取る（ポエムや独り言に近いリズム）
- 結論を出そうとせず、思考の過程そのものを記録する
- 過去の自分との対話や、現在の心情の吐露が中心

**特徴的なパターン**:

- 日常の出来事→そこから派生した内省・思考
- 自問自答（「〜だったんだっけ？」「〜なんだろうか」）
- 心の声の描写（「〜という心の声と〜という心の声が戦っている」）
- 文末に「〜なあ。」「そんなところ。」で締めることがある

**例**:

```text
本当は、好きを共有してより楽しむことができると幸せなのかもしれないけど、
好きという気持ちは、自分だけのものという感覚が大きいのかもしれない。
```

---

### 2. 技術記事 (Technical Articles)

**タグ**: `[技術]`（外部公開リンクが含まれる）

**文体**:

- 「です・ます」調
- 丁寧で説明的
- 一人称は「筆者」「私」

**構成**:

1. **導入**: 問題提起や背景（「〜する方法を紹介します」）
2. **注意書き**: `:::message alert` 等を用いた免責事項
3. **手順/解説**: 見出し（`##`）で明確にステップを分ける
4. **コードブロック**: 言語名を明記（`bash:WSL`など）、ファイルパスやDiffも見やすく
5. **結果/まとめ**: 成果を簡潔に

**特徴**:

- 読者（他人）を意識した書き方
- 「筆者も解決したので〜」と同じ境遇の人への共有というスタンス
- 参考元のリンクを明記

**例**:

```markdown
## 手順1: `antigravity` コマンドへのパスを通す

そもそもWSL上で `antigravity` コマンドが見つからない場合、まずはパスを通す必要があります。
```

---

### 3. 開発日記 / 技術メモ (Dev Diary / Technical Notes)

**タグ**: `[技術, 日記]`（両方を併用）

**文体**:

- 「だ・である」調
- 技術的な内容だが、自身の理解を確認するための口調
- 「〜らしい」「〜なんだろう」「〜かもしれない」

**構成**:

- その日やったことの報告
- やりたいこと / わかったこと / 不明点のリスト化
- GitHubリンクやIssue番号を積極的に挿入
- 章立てて深掘りする部分と、雑記部分の混在

**特徴**:

- 「完璧な解説」より「現時点での理解」を優先
- 未解決事項もそのまま書く（「〜は不明」「うまくいっていない」）
- AIに任せた部分と自分で考えた部分の区別を書くことがある

**例**:

```text
AIに雑に渡しても、ちゃんと調査せず、とりあえず動かしてe2eテストを流す。
みたいなループをしまくっていて効率が悪い。
```

---

### 4. 読書メモ / レビュー (Reviews / Essays)

**タグ**: `[読書]`, `[日記]`

**文体**: 「だ・である」調

**構成**:

1. **導入**: なぜそれを手に取ったか、どういう状況か
2. **対象情報**: 本や作品のリンク（Amazon等）
3. **感想**: ネタバレを配慮しつつ、自分にどう刺さったか

**特徴**:

- 客観的な評価よりも、**主観的な体験**を重視
- 自分の生活や過去の経験との照らし合わせ
- 「自分には響かない」「〜が気になった」などの正直な感想
- 「買って読んでみたい」と思わせる情報にしたいという意識

**例**:

```text
「すごいとは思うが、自分には響かない」といった複雑な感情を抱いた。
```

## 文体・表現の特徴 (Writing Style)

### 推測・保留の表現

断定を避け、思考の揺らぎを残す傾向がある。

- 「〜気がしている」
- 「〜なのかもしれない」
- 「〜だろう」
- 「〜だと思う」
- 「〜らしい」

### 接続・転換

一度書いたことに対して、逆の視点や深掘りを入れる際に使用。

- 「ただ、」
- 「しかし、」
- 「とはいえ、」
- 「やっぱり、」
- 「そもそも」
- 「そんなわけで」
- 「まあ」

### 自問自答

思考の流れを可視化する表現。

- 「〜だったっけ？」
- 「〜なんだろうか」
- 「〜なんだろうな」
- 「で、今日の日記は、なにを書こうと思ってたんだっけ？」

### 心の声の描写

内面の葛藤を表現する手法。

- 「〜という心の声と、〜という心の声が激しく戦っている」
- 「勝手な妄想が進んでしまう」

### 感情・感覚の表現

感覚的な表現を擬音や俗語を交えて使う。

- 「なんか良かった」
- 「ホッとした」
- 「刺さった」
- 「難儀な性格」
- 「流石に」
- 「むずむずしてしまう」

### カジュアルな口語表現

- 「超うめぇ」
- 「チマチマ」
- 「ぼちぼち」
- 「ガッツリ」
- 「さすがに」
- 「〜なあ」（文末）

### 比喩・ユーモア

独特の表現や自己ツッコミ。

- 「馬鹿のリリースノート」
- 「獣道すら無い山道を登っているような感覚」
- 「DevOpsからOps引いたらただのデブじゃん」
- 「おもちゃを与えられた子どもの如く」
- 「乗れる巨人の肩があるなら乗りたい精神」

### 自己ツッコミ

客観的に自分を見る視点。

- 「難儀な性格だと思う」
- 「こんな人間になってないし」
- 「我ながら〜」
- 「最後になんとも他人事だ…」

## 内省的な傾向 (Introspective Style)

### 思考過程の記録

- 行動の報告より「なぜそう思ったか」「どう感じたか」に焦点
- 結論を出さず、思考の途中を残すことを良しとする
- 未解決・未完結を許容する姿勢

### 過去の自分との対話

- 過去に書いた文章を引用・参照する
- 「〜年前の自分が〜」という振り返り
- 記録することの意義への言及

### 承認欲求への自覚

- 「見られることを意識して」という意識
- 「スキがついた通知が来ると、『お、やったぜ！』となっている」
- 承認欲求と向き合う正直さ

## PS（追伸）の使い方

本文の後に `PS.` として追記することがある。本文に入れづらい補足や、ふと思ったことを書く。

```text
PS. 生成AIに添削してもらっているけど、どれくらい生成AIっぽい文章になっているんだろう。
自分じゃわからない。
```

## 執筆時のスタンス

1. **他人が面白いと思うことを狙って書かない**
2. **自分のための文章を書く**
3. **わかりやすく書くことを意識する**（人の目を意識することで文章力向上を図る）
4. **未来の自分に見せるため**の記録
5. **実時間をより少なく、情報量を多く**（生活ログより深い内省）
